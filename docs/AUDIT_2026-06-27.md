真实 Legado 书源端到端跑不通。

| 能力 | 状态 | 对标 Legado | 能跑通真实 Legado? |
| --- | --- | :---: | :---: |

| URL 请求构建 | 零实现，消费 host 预取 response | 0% | ❌ |

| 远程阅读纵切 | dispatch 接线完成，因三重缺失跑不通真实书源 | ~30% | ❌ |
| 数据持久化 | 零 SQLite，仅 `InMemoryStorage` | 0% | ❌ |
| 本地书 | 仅 TXT 真实；EPUB/PDF/Mobi/Umd 只有能力声明 | ~20% | ❌ |

## 4. iOS（`Reader for iOS`）审计

对标 Legado 约 50%。三平台中最完整，但业务绑死旧 Swift Core。

- **业务来源**：84 文件 `import ReaderCoreModels/Parser/Protocols`（旧 Swift Core），
  仅 9 文件接入 Rust Core（adapter 壳 / evidence / debug，非业务路径）
- **UI**：35 个 SwiftUI 页面（书架/搜索/详情/目录/正文/阅读器/书源管理/RSS/WebDAV/文件导入/Debug），三平台最完整
- **持久化**：完整 Persistence 层（Swift 持久化，非 Rust Core）
- **WebDAV**：真实（`URLSessionWebDAVClient` 真发 PROPFIND/PUT/GET/DELETE）
- **TTS**：真实（`AVSpeechSynthesizer` + delegate + 播放控制）
- **网络**：`URLSessionHTTPClient` + `LiveFetchExecutor`（真实网络）
- **WebView**：`ProductionWebViewAdapter` + `WebViewSecurityGate` + Cookie mirror
- **Rust Core 接入**：真 C ABI 调用（`rc_runtime_create/send/cancel/destroy`），
  但业务层用 `MockReaderCoreService`，真实与 mock 并存，业务路径未切到 Rust Core

核心矛盾：依赖旧 Swift Core，Rust Core 接入仅 adapter 壳。

## 5. Android（`Reader for Android`）审计

对标 Legado 约 20%。底层最厚但 UI 归零，Core 假接入，业务全独立。

- **业务来源**：全部独立自实现（绕过 Core）——`RuleExpression` + `JsoupMarkupParserAdapter`
  自实现规则引擎，`SearchParser`/`ContentParser`/`TOCParser`/`BookInfoParser` 独立，
  `HttpClient`/`OkHttpAdapter` 独立请求层
- **UI**：完全删除（零 Activity/Composable/Fragment）
- **持久化**：三平台最完整——Room + KSP 生成（Bookmark/ReadingProgress/CachedChapter/
  SyncOperationLog/AppDatabase）+ DataStore + 配置实体
- **WebDAV**：内存 mock（`WebDavClient` 用 `files: Map` 模拟 PROPFIND，非真实网络）
- **TTS**：真实队列逻辑（`TtsQueue` + `AndroidTtsAdapter` + `ChapterTextFeeder`）
- **Rust Core 接入**：假桩——`reader_native_runtime_evidence.cpp` 不链接 Rust Core，
  只解析 JSON 返回 identity/`hostBusLoopDispatched`；`RealCoreBridge` 名不副实

核心矛盾：UI 归零 + Core 假桩 + 业务全独立，是 strangler migration 最大上收对象。

## 6. HarmonyOS（`Reader for HarmonyOS`）审计

对标 Legado 约 30%。平台 adapter 最全，UI 最简，NAPI 是 POC。

- **业务来源**：Facade 层自实现 + NAPI POC——`RulePipelineFacade` 自实现规则求值，
  `MarkupParserAdapter` 自实现 HTML/XML 解析，`SearchService` 标注
  `BLOCKED_BY_BRIDGE_RUNTIME` 走 fixture fallback
- **UI**：仅 5 个页面（Index/Bookshelf/Search/Reader/Settings），最简壳
- **平台 adapter**：三平台最全——14 个（ArkWeb/Cookie/Download/FilePicker/HuksCredential/
  PdfRender/SystemTts/HttpTts/AudiobookPlayer/WebDavClient/ZipArchive/CBZParser/
  Deflate/RawDeflate）
- **持久化**：`StoreFacade` 是 KV 内存 map（fixture 级，非真实 SQLite）
- **TTS**：最全（SystemTts + HttpTtsContract + AudiobookPlayer，唯一覆盖 HttpTTS）
- **本地书**：格式覆盖最广（PDF/CBZ/Zip/TXT/EPUB contract）
- **Rust Core 接入**：真 NAPI 导入（`import libreader_core_napi.so`），但注释标明
  `C ABI/NAPI 为长期可选轨道，先做 POC`，有真机 `.so` artifact，无 real-device proof

核心矛盾：UI 极简 + NAPI 未出 POC + 持久化 fixture 级。

## 9. 关键发现

1. **Core 未成为任何平台的业务来源**——iOS 用旧 Swift Core，Android 自实现，
   HarmonyOS Facade 自实现。三方各自为政，能力重复且不一致。
2. **原 Swift Core 是可靠迁移源但本身也未完备**——规则引擎/URL/JS/WebDAV/RSS/本地书
   可迁；TTS/主题/SQLite/TxtTocRule/Bookmark 需对照 Legado 新建。
3. **三平台处于完全不同阶段**，不能统一处理——iOS UI 最完整但 Core 用 mock；
   Android 底层最厚但 UI 归零；HarmonyOS adapter 最全但 UI 最简。
4. **UI 层是共同空缺**——UI 仓库是 HTML 原型，iOS 有原生 UI 但绑旧 Core，
   Android 无 UI，HarmonyOS 5 页壳。
5. **真实 Legado 书源在 Rust Core 端到端跑不通**，阻塞在 AnalyzeUrl + 完整 DSL +
   JS 网络类三重缺失，这是 S4 远程阅读纵切的前置。
